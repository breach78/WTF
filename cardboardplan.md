# 인덱스 카드 모드 구현 플랜

## 1. 목적

이 플랜의 목표는 현재 `트리 기반 카드 편집기` 위에 `Card Buddy` 계열의 인덱스 카드 보드 모드를 추가하는 것이다.

핵심 기준:

- 카드 이동과 재배치는 처음부터 매우 부드러워야 한다.
- 인덱스 카드 모드는 새 데이터 모델이 아니라 `기존 트리의 다른 투영(view projection)`이어야 한다.
- 현재 앱의 `스플릿`, `포커스 모드`, `언두`, `히스토리`, `AI 패널`, `검색`과 충돌하지 않아야 한다.
- v1은 완전 자유 배치보다 `카드 중심 flat board surface`에 집중한다.
- 부모 구조는 유지하되, 부모 카드는 더 이상 큰 패널로 전면에 나오지 않는다.
- 창작 흐름을 방해하는 미세한 안내, 경고, 강조는 넣지 않는다.

## 2. 재설계 원칙

이번 재설계의 핵심은 `group wrapper 중심 화면`을 `flat board surface 중심 화면`으로 바꾸는 것이다.

즉:

- 저장 구조를 새로 만들지 않는다.
- 세션 상태를 새로 만들지 않는다.
- 큰 카드 시트, 요약, Temp, 검색, AI, 언두/히스토리는 최대한 재사용한다.
- 버리는 것은 주로 `렌더링`, `레이아웃`, `드래그` 계층이다.

재사용 대상:

- 기존 `SceneCard` 트리 구조
- 기존 언두/리두 경로
- 기존 히스토리 기록 경로
- 기존 큰 카드 시트 편집 진입
- 기존 summary sidecar
- 기존 Temp 카드 저장 위치
- 기존 검색/AI/스플릿 연결

교체 대상:

- `IndexBoardProjection -> groups` 중심 보드 투영
- 그룹 박스 렌더링
- 그룹 내부 2열 그리드 레이아웃
- 그룹 박스 드래그와 그룹 reorder UI
- 패널 중심 헤더/시각 시스템

## 3. 모드 정체성

인덱스 카드 모드는 포커스 모드처럼 카드 한 장을 깊게 보는 모드가 아니다.
이 모드는 `한 컬럼 전체를 다른 방식으로 편집하는 pane-level 작업 모드`다.

- 같은 `ScenarioWriterView` 안에서 동작한다.
- 현재 pane의 메인 작업창을 인덱스 카드 보드로 전환한다.
- 포커스 모드처럼 전체를 덮는 오버레이가 아니라, `기본 캔버스 교체`에 가깝다.
- 진입/복귀 시 선택 카드, 스크롤, 편집 상태 복원은 포커스 모드처럼 정교하게 유지한다.
- 같은 pane에서 인덱스 카드 모드가 열리면 일반 작업창은 렌더 트리에서 빠지고 사실상 멈춘다.
- 즉, 인덱스 보드 퍼포먼스에 일반 작업창 레이아웃/스크롤/측정 비용이 영향을 주지 않게 한다.

## 4. 스플릿과의 관계

- 스플릿에서는 `한쪽 pane = 인덱스 카드`, `다른 쪽 pane = 일반 작업창 또는 포커스 모드`를 허용한다.
- v1에서는 `한 시나리오당 활성 인덱스 보드 세션은 1개`로 제한한다.
- v1에서는 양쪽 pane을 동시에 인덱스 보드로 여는 것은 막는다.
- 같은 pane에서 `인덱스 카드 모드`와 `포커스 모드`를 동시에 활성화하지 않는다.

## 5. 진입 방식

- 컬럼의 우클릭 메뉴에 `인덱스 카드 뷰로 보기`를 둔다.
- 카드 우클릭 메뉴에도 같은 액션을 둔다.
- 사용자가 선택한 `컬럼`은 화면상 컬럼 번호가 아니라, 실제로는 `그 컬럼을 형성하는 parent card ID`로 식별한다.
- 따라서 보드 세션은 `scenarioID + sourceParentID + sourceDepth` 기준으로 열고 복원한다.

## 6. 보드의 기본 구조

### 6-1. 핵심 전환

이제 보드는 `group wrapper`를 큰 박스로 렌더링하지 않는다.

대신:

- 부모 카드는 `lane` 메타데이터로 남는다.
- 실제 화면의 1차 객체는 항상 `카드`다.
- 부모 정보는 `얇은 lane label` 또는 `작은 칩`으로만 노출한다.
- 사용자는 보드에서 먼저 카드 덩어리를 읽고, 필요할 때만 부모 정보를 읽는다.

### 6-2. 새 투영 구조

보드 전용 투영은 `IndexBoardProjection -> groups` 대신 `BoardSurfaceProjection -> surfaceItems` 중심으로 다시 잡는다.

권장 구조:

```swift
struct BoardSurfaceProjection {
    let source: IndexBoardColumnSource
    let lanes: [BoardSurfaceLane]
    let surfaceItems: [BoardSurfaceItem]
    let orderedCardIDs: [UUID]
}

struct BoardSurfaceLane {
    let parentCardID: UUID?
    let laneIndex: Int
    let labelText: String
    let subtitleText: String
    let colorToken: String?
}

struct BoardSurfaceItem {
    let cardID: UUID
    let laneParentID: UUID?
    let laneIndex: Int
    let slotIndex: Int
}
```

의미:

- `lane`은 부모 구조를 유지하기 위한 메타데이터다.
- `surfaceItem`은 실제 보드에서 움직이는 카드 단위다.
- 카드가 먼저 있고, lane 정보는 카드에 부착된 분류 정보처럼 작동한다.

### 6-3. 부모 정보 표현 방식

부모 카드는 일반 카드처럼 평면에 섞이지 않는다.
하지만 더 이상 큰 패널도 아니다.

표현 원칙:

- lane 시작 카드 위에 작은 칩으로 부모 제목을 표시한다.
- 필요하면 lane의 첫 슬롯 근처에 얇은 label row를 둔다.
- 부모의 상태/짧은 내용은 칩 안의 짧은 텍스트나 툴팁 수준으로만 노출한다.
- lane label은 보드를 읽는 힌트여야지, 카드보다 시선을 더 가져가면 안 된다.

## 7. 레이아웃 원칙

### 7-1. 전체 구조

v1은 여전히 자유 배치가 아니라 `강한 자동 정렬 보드`로 간다.
다만 정렬의 단위가 `그룹 내부 그리드`가 아니라 `보드 전체 슬롯 매트릭스`다.

- 카드의 기본 크기와 글자 크기는 안정적으로 유지한다.
- 시야를 더 크게 보거나 더 작게 보는 문제는 카드 재사이즈보다 `보드 줌`으로 해결한다.
- 카드 크기는 균일하다.
- 카드들은 모두 같은 전역 슬롯 체계에 배치된다.
- lane은 시각적 메타데이터이지, 별도 박스가 아니다.

### 7-2. Global Slot Layout

이 레이아웃의 핵심은 `전역 슬롯 매트릭스`다.

- 각 카드는 `slotIndex`를 가진다.
- 슬롯은 보드 전체에서 row-major 또는 column-aware 방식으로 계산한다.
- 드래그 중인 카드만 포인터를 따라 1:1로 움직인다.
- 나머지 카드는 `slot reassignment`로 부드럽게 미끄러진다.
- placeholder는 개별 그룹 안이 아니라 전역 슬롯 위에서 유지된다.
- 사용자는 항상 `카드 표면 전체가 재정렬되는 느낌`을 받아야 한다.

즉, 구현 중심은 다음과 같다.

- 고정 슬롯 계산
- 드래그 중 placeholder 슬롯 유지
- 실제 카드 렌더는 transform/offset 기반 애니메이션
- 드래그 카드만 absolute overlay
- 나머지 카드는 slot change만 반영

### 7-3. Lane과 슬롯의 관계

lane이 있다고 해서 lane마다 별도 그리드를 만들지 않는다.

원칙:

- lane은 카드 정렬의 힌트이자 정렬 기준이다.
- 같은 lane의 카드들은 보드 위에서 연속된 슬롯 구간을 우선 배정받는다.
- 하지만 드래그 엔진은 전체 보드 슬롯을 하나의 표면으로 취급한다.
- 따라서 카드 이동 감각은 전체 표면 기준으로 유지되고, 부모 구조는 드롭 시점의 lane/인접 카드 해석으로 반영된다.

### 7-4. 보드 bounds

- 보드는 `무한 캔버스`가 아니라 현재 카드가 놓인 범위를 감싸는 `유한 보드`로 잡는다.
- 보드 범위는 `현재 카드 bounds + 넉넉한 상하좌우 작업 여백`으로 계산한다.
- 이 여백은 카드 추가, 재배치, Temp 카드 투입을 위한 공간으로 사용한다.
- 사용자는 빈 공간 속에서 길을 잃는 것이 아니라, 항상 `카드 덩어리가 중심인 보드`를 보게 해야 한다.

## 8. 카드 인터랙션

### 8-1. 카드 편집

v1에서는 보드 카드 위에서 직접 장문 편집하지 않는다.

- 보드에서는 선택, 이동, lane 읽기, 빠른 훑어보기 중심
- 본문 편집은 `같은 pane 위에 뜨는 큰 카드 시트`에서 수행

### 8-2. 큰 카드 시트

Card Buddy 계열 UX 기준으로, 보드 뒤에 어두운 dimmed background를 깔고 가운데 큰 카드 편집 시트를 띄운다.

- 보드는 뒤에 그대로 남는다.
- 사용자는 보드 문맥을 잃지 않는다.
- 시트에서 저장 후 다시 보드로 복귀한다.
- v1에서는 이 시트가 인덱스 카드 모드의 유일한 정식 편집 진입점이다.
- 큰 카드 시트가 열려 있는 동안 보드의 드래그/재배치 인터랙션은 멈춘다.

열기 규칙:

- 카드 더블클릭
- Enter
- 카드의 작은 확대 아이콘 클릭

### 8-3. 카드 앞면/뒷면

카드는 `앞면 = 원문`, `뒷면 = 짧은 요약` 구조를 갖는다.

- 카드 앞면은 원문 일부만 보여준다.
- 카드 뒷면은 1줄 또는 2~3줄의 짧은 요약만 보여준다.
- 카드에 `요약면 있음`을 나타내는 작은 배지 아이콘을 둔다.
- 사용자가 카드를 뒤집어 놓은 상태에서도 정렬과 이동은 동일하게 동작해야 한다.
- 큰 카드 시트 안에서도 앞면/뒷면 전환을 지원한다.

## 9. 드래그와 실제 트리 반영

### 9-1. 카드 이동

보드에서 한 변경은 실제 데이터에 즉시 반영한다.
단, 복잡한 조상 재구성을 완벽하게 계산하려고 하지 않는다.

원칙:

- 카드 이동은 드롭 완료 시 바로 실제 `parent/orderIndex`를 갱신한다.
- 드래그 중간에는 데이터 저장이나 모델 커밋을 하지 않는다.
- placeholder와 slot reassignment는 전적으로 surface layer에서 처리한다.
- 드롭이 끝난 뒤 한 번만 실제 트리에 반영한다.

### 9-2. 유효 parent 결정

유효 parent 결정은 기존 철학을 유지한다.

우선순위:

1. 드롭 위치 바로 앞 카드의 parent
2. 드롭 위치 바로 뒤 카드의 parent
3. 앞 카드의 조상 중 첫 유효 parent
4. 뒤 카드의 조상 중 첫 유효 parent
5. 그래도 없으면 최상위 안전 parent

이 방식의 목표:

- 보드에서 만든 흐름이 실제 트리 순서에 즉시 반영된다.
- 복잡한 grandparent 재구성 로직 없이도 실사용이 가능하다.
- 구조가 아주 완벽하지 않아도 사용자는 작업창에서 빠르게 후정리할 수 있다.

### 9-3. Lane reorder

그룹 이동이 꼭 필요하다면 `큰 패널 드래그`가 아니라 `lane label reorder`로 간다.

- lane label은 드래그 가능할 수 있다.
- lane reorder는 실제로는 해당 부모 계열 카드 묶음의 우선 순서를 바꾸는 액션이다.
- 사용자가 lane label을 움직이면, 해당 lane의 카드 시작점과 lane index가 재계산된다.
- 카드 표면은 여전히 큰 박스 없이 유지된다.

v1 원칙:

- 그룹 패널 드래그는 하지 않는다.
- 큰 컨테이너 자체를 옮기는 느낌은 제거한다.

### 9-4. 미세 알림 배제

창작 흐름을 방해하지 않도록 다음은 v1에서 의도적으로 넣지 않는다.

- 드롭 직전 구조 설명 문구
- `임시 배치` 같은 약한 배지
- 작업창 복귀 시 방금 재배치된 카드 강조
- 구조가 애매하다는 시각적 경고

보드에서는 그냥 옮기고, 실제 트리는 즉시 바뀌며, 사용자는 나중에 작업창에서 정리한다.

## 10. 카드 시각 시스템

레퍼런스에 가까워지려면 카드 스타일을 다시 잡아야 한다.

원칙:

- 카드 반경은 더 작게 간다.
- 그림자는 더 얕고 넓게 간다.
- 카드 내부 패딩은 줄이고, 텍스트 밀도는 높인다.
- 카드 색상은 더 분명하게 분류한다.
- 회색 카드 일색 구조를 줄인다.
- active/selected 상태는 두껍고 무거운 보더보다 얇고 정확한 강조로 해결한다.

목표 인상:

- 화면 전체를 빠르게 스캔할 수 있어야 한다.
- 카드가 `편집용 위젯`보다 `정리된 메모 카드`처럼 보여야 한다.
- 카드가 많아져도 읽는 압박보다 흐름이 먼저 느껴져야 한다.

## 11. 요약 전략

v1 요약은 `원문에 섞어 저장하지 않는다`.

- 원문 `content`는 그대로 둔다.
- 요약은 별도 sidecar 데이터로 관리한다.
- 초기에는 기존 AI digest의 `shortSummary`를 재사용해도 된다.
- 이후 AI 1줄 요약, 수동 요약 편집으로 확장 가능하다.

권장 파일:

- `card_summaries.json`

권장 레코드:

- `cardID`
- `summaryText`
- `sourceContentHash`
- `updatedAt`
- `sourceType` (`digest`, `ai`, `manual`)
- `isStale`

## 12. Temp 카드

### 12-1. 목적

사용자가 즉석 아이디어를 잃지 않게, 보드의 빈 공간에서 바로 새 카드를 만들 수 있어야 한다.

### 12-2. 저장 위치

이 카드는 `floating`으로 만들지 않는다.

저장 위치는 반드시:

- `root`
- `노트`
- `temp`

아래의 하위 카드여야 한다.

이유:

- `floating` 카드는 루트/트리 투영에서 빠진다.
- 현재 AI와 여러 전역 집계도 floating 카드를 기본 제외한다.
- 임시 아이디어도 결국 시나리오 구조 안에 남아야 한다.

### 12-3. 생성 규칙

- 보드 빈 배경 더블클릭
- 또는 `N`
- 또는 `+ Temp 카드`

로 생성한다.

생성 흐름:

1. `root -> note -> temp` 경로를 찾는다.
2. 없으면 `temp` 컨테이너 카드를 자동 생성한다.
3. 새 임시 카드를 `temp`의 마지막 자식으로 추가한다.
4. 즉시 큰 카드 시트를 열어 입력하게 한다.

보드에서는 이들을 별도 큰 박스가 아니라 `Temp lane` 또는 `Temp chip`으로 보여준다.

## 13. 줌과 캔버스 탐색

### 13-1. 핀치 줌

인덱스 카드 보드는 반드시 핀치 줌을 지원한다.

원칙:

- 일반 작업창의 `mainWorkspaceZoomScale`과 분리된 `indexBoardZoomScale`을 둔다.
- 보드 위에서 핀치하면 포인터 기준으로 확대/축소한다.
- 보드 전체가 작게 멀어지거나 크게 가까워지는 감각을 유지한다.
- 줌값은 세션별로 기억한다.

부가 기능:

- 버튼으로 `100%` 복귀
- `Cmd + / Cmd - / Cmd 0`
- 트랙패드 핀치

### 13-2. 스크롤

- 2축 스크롤을 지원한다.
- 확대 상태에서도 drag moving과 panning을 분리해야 한다.
- 카드 드래그 중에는 보드가 가장자리에서 자동 스크롤될 수 있어야 한다.
- 보드의 스크롤 가능 범위는 `무한`이 아니라 현재 콘텐츠 bounds와 작업 여백 안으로 제한한다.
- 카드나 lane label이 보드 가장자리에 가까워지면, 그 방향의 보드 bounds를 조금씩 확장해 작업이 막히지 않게 한다.

## 14. 헤더와 상단 크롬

헤더는 많이 줄여야 한다.

원칙:

- 상단 제목/설명은 한 줄 수준으로 압축한다.
- 보드 설명 문구는 2줄 이상 차지하지 않는다.
- 줌, Temp, 복귀는 우상단의 얇은 컨트롤 바로 정리한다.
- 보드가 화면의 주인공이어야 하고, 헤더는 배경으로 물러나야 한다.

즉:

- 현재의 무거운 정보 블록형 헤더는 피한다.
- 제목은 유지하되, 구조 설명은 최소화한다.
- 버튼 스타일도 테두리/높이를 줄여 시각적 소음을 낮춘다.

## 15. 기존 시스템과의 통합

### 15-1. 언두/리두

데이터 변화는 기존 `ScenarioState` 기반 언두를 그대로 사용한다.

언두 대상:

- 카드 재정렬
- lane reorder
- lane 간 카드 이동
- 큰 카드 시트에서의 본문 편집
- Temp 카드 생성
- Temp lane 자동 형성

언두 비대상:

- 카드 뒤집기
- 카드 시트 열기/닫기
- 보드 줌
- 보드 스크롤
- 선택 강조
- AI 패널 열기/닫기
- 검색어 입력

요약 sidecar는 v1에서 파생 캐시로 보기 때문에 기본적으로 언두 대상에서 제외한다.

### 15-2. 히스토리

실제 카드 구조/본문 변경은 기존 히스토리 스냅샷 체계에 그대로 반영한다.

원칙:

- 인덱스 보드에서 일어난 카드 이동/편집도 히스토리에 기록한다.
- v1에서는 인덱스 보드 전용 히스토리 UI를 만들지 않는다.
- 히스토리 버튼을 누르면 기존 히스토리 UI로 전환하고, 닫으면 직전 보드 세션으로 복귀한다.
- 카드 드래그 중간 상태는 히스토리에 남기지 않고, 드롭 확정 상태만 남긴다.

요약 sidecar는 v1에서 히스토리 대상에서 제외한다.

### 15-3. 오른쪽 검색 패널

기존 timeline/search 패널은 유지한다.

동작 원칙:

- 인덱스 보드에서도 오른쪽 검색 패널을 열 수 있다.
- 검색 결과가 현재 보드 범위 안이면 보드에서 선택하고 해당 카드가 보이는 위치로 이동한다.
- 검색 결과가 현재 보드 범위 밖이면:
  - 스플릿 활성 시 반대 pane의 일반 작업창으로 연다.
  - 싱글 pane이면 인덱스 보드를 빠져나와 일반 작업창으로 점프한다.

### 15-4. AI 패널

AI 패널은 유지한다.

원칙:

- 인덱스 보드의 선택 카드와 active card를 그대로 AI scope에 연결한다.
- AI 질문은 현재 선택 카드 또는 활성 카드 기준으로 작동한다.
- v1에서는 AI가 구조를 크게 바꾸는 액션은 제한한다.

허용:

- 활성 카드 본문에 답변 붙이기
- 선택 카드 문맥 상담

보류:

- 하위 카드 자동 생성
- lane 구조 자동 재편

추가 규칙:

- `노트 전체` AI 범위에서 `temp` 하위 카드는 기본 제외하는 옵션을 둔다.

## 16. 인덱스 보드와 포커스/작업창의 관계

### 16-1. pane 내 상태 머신

한 pane은 다음 중 하나의 기본 모드만 가진다.

- 일반 작업창
- 포커스 모드
- 인덱스 카드 보드

큰 카드 시트는 별도의 `overlay editor state`다.

즉:

- `보드 + 큰 카드 시트` 가능
- `보드 + 포커스` 불가
- `일반 작업창 + 포커스` 기존 유지

추가 원칙:

- 인덱스 카드 보드가 활성인 동안 같은 pane의 일반 작업창은 숨기는 수준이 아니라 사실상 정지 상태여야 한다.
- 포커스 모드처럼 `opacity 0`으로 남겨 두지 않고, 상위 분기에서 아예 다른 뷰로 교체한다.
- 모델 관찰은 유지하되, 일반 작업창의 레이아웃 계산과 geometry 관찰은 중단한다.

### 16-2. 스플릿 조합

허용 조합:

- 좌 인덱스 / 우 작업창
- 좌 인덱스 / 우 포커스
- 좌 작업창 / 우 인덱스
- 좌 포커스 / 우 인덱스

v1 금지:

- 좌 인덱스 / 우 인덱스

## 17. 성능 목표

인덱스 카드 모드는 성능이 미완이면 실패다.

최소 목표:

- 드래그 중 체감 끊김이 없어야 한다.
- 전역 슬롯 결정이 흔들리지 않아야 한다.
- 카드 수가 많은 상태에서도 스크롤과 드래그가 즉시 반응해야 한다.
- 확대/축소/드래그/오토스크롤이 동시에 일어나도 프레임 드랍이 심하지 않아야 한다.

기술 원칙:

- `projection`, `layout`, `render`, `commit` 계층을 분리한다.
- 슬롯 계산은 데이터 구조로 유지한다.
- 카드 뷰는 최소 invalidation으로 간다.
- 드래그 중 full relayout을 최소화한다.
- 드래그 중 세션 저장이나 모델 저장을 하지 않는다.
- 저장과 모델 커밋은 드래그 완료, 편집 완료 같은 commit 시점에만 수행한다.
- 필요 시 계측 로그와 diagnostics를 추가한다.

## 20. 최종 판단

이 기능의 본질은 `트리 구조를 버리는 것`이 아니라, `트리 구조를 카드 중심 surface로 재투영하는 것`이다.

따라서 구현의 중심은 네 가지다.

- 구조적으로는 `기존 트리/언두/히스토리/AI/세션`을 그대로 활용한다.
- 투영적으로는 `BoardSurfaceProjection + lane metadata + surfaceItems`로 다시 잡는다.
- UX적으로는 `flat board surface + 같은 pane 큰 카드 시트 + 앞/뒤 카드`를 만든다.
- 성능적으로는 `드래그 중 부드러움`을 첫 번째 품질 기준으로 삼는다.

그리고 v1의 중요한 운영 원칙은 다음 세 가지다.

- 보드에서의 변경은 실제 트리에 즉시 반영하되, 애매한 경우는 가장 가까운 유효 parent 아래에 붙인다.
- 저장과 기록은 드래그 중간이 아니라 `이동 완료`, `편집 완료` 같은 명확한 commit 시점에서만 한다.
- 부모 정보는 유지하되, 더 이상 큰 박스로 전면에 세우지 않고 lane label/chip 수준으로만 드러낸다.

이 세 가지를 지키면 인덱스 카드 모드는 현재 앱의 자연스러운 확장으로 붙으면서도, 레퍼런스에 훨씬 가까운 화면과 감각을 만들 수 있다.
